Here's the complete flow in plain English — think of it as a conversation between 4 parties: Your User, Your React App, Your FastAPI Backend, and Google.

The "Connect Gmail" Flow
Phase 1 — User Initiates
The user is already logged into your app (they have your app's JWT token)

They click "Connect Gmail" button in React

React calls your FastAPI backend: "Hey, generate a Google login URL for this user"

FastAPI creates a Google OAuth URL and secretly embeds the user's ID inside it (this is the state parameter — it's just a way to remember who started this)

FastAPI sends that URL back to React

React redirects the user's browser to Google's login page

Phase 2 — User Consents on Google
The user sees Google's consent screen: "Your App wants to access your Gmail — Allow or Deny?"

User clicks Allow

Google generates a one-time authorization code (valid for ~10 minutes, single-use)

Google redirects the user's browser back to your FastAPI callback URL, attaching that code + the state (user ID) in the URL

Phase 3 — Backend Exchanges Code for Tokens
FastAPI receives the callback — it reads the state to know which user just connected

FastAPI takes the authorization code and calls Google's token endpoint directly (server-to-server, user is not involved)

Google validates the code and responds with two tokens:

Access Token — short-lived (1 hour), used to actually call Gmail API

Refresh Token — long-lived, used to silently get new access tokens when they expire

FastAPI encrypts both tokens and saves them in your database, linked to that user's ID

FastAPI redirects the user back to your React dashboard — "Gmail connected successfully!"

Phase 4 — Agent Uses Gmail on Behalf of User
When the user runs the AI agent, FastAPI fetches that user's tokens from the DB

If the access token is expired, FastAPI quietly calls Google with the refresh token to get a new one — the user never sees this

The Gmail API tools use the access token to read/send emails

Google responds with the email data — all scoped to only that user's inbox

The Key Insight
The authorization code is like a one-time voucher Google gives the user after they consent. Your backend immediately exchanges it for real tokens before it expires. From that point on, your backend never asks the user to log in to Google again — it silently refreshes the access token using the refresh token whenever needed. The user connects once, and the agent works forever until they revoke access.
